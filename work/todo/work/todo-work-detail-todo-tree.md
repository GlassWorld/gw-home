# 업무 상세 연관 TODO 트리 기능 계획

## Goal

- 목적: 업무 상세에 부모/자식 구조의 연관 TODO 트리를 추가하고, 진행률·상태·일정·정렬·접기/펼치기 기능을 포함한 신규 메뉴 `업무 세부 작업` 화면을 제공한다.

## Scope

- 포함:
  - `work` 도메인 TODO 전용 테이블 및 관련 DDL 추가
  - TODO `VO`, DTO, Mapper, Service, Controller 추가 또는 확장
  - 업무 상세 조회 응답에 TODO 트리 및 집계 정보 포함
  - 신규 메뉴 `업무 세부 작업` 추가
  - TODO 트리 조회/추가/수정/삭제/완료 처리 UI 구현
  - 완료 항목 숨기기, 접기/펼치기, 들여쓰기 표시, 상태/일정 표시 구현
- 제외:
  - 다른 도메인과의 크로스 도메인 연동
  - 대시보드 전체 개편
  - 간트차트, 드래그앤드롭 정렬 같은 확장 UI
  - 업무 목록 전체를 TODO 중심 화면으로 전면 재구성하는 작업

## Current State

- 현재 상태:
  - `WorkUnit` 은 업무 기본정보와 Git 프로젝트 연결만 관리한다.
  - 업무 상세 API는 TODO 트리 데이터를 제공하지 않는다.
  - 프런트 `/work` 화면은 업무 목록/등록 관리에 집중되어 있다.
  - 내비게이션에 `업무 세부 작업` 메뉴는 아직 없다.
- 제약:
  - DB 는 MyBatis + 수동 DDL 규칙을 따라야 한다.
  - 외부 API 식별자는 `uuid` 만 사용해야 한다.
  - 상위 TODO 상태와 진행률은 직접 수정이 아니라 하위 항목 기준으로 일관되게 계산되어야 한다.
  - 기존 `업무관리` 화면 동작은 깨지지 않아야 한다.

## Action Items

1. 데이터 모델과 상태 규칙을 정의한다.
   - `tb_wrk_todo` 계열 테이블 구조를 설계한다.
   - `workUnitId`, `parentId`, `title`, `description`, `status`, `progressRate`, `startDate`, `dueDate`, `sortOrder`, `depth` 또는 조회용 계층 정보 필요 여부를 정한다.
   - leaf 완료, 상위 진행률, 지연 상태 계산 규칙을 확정한다.
2. 백엔드 API 를 구현한다.
   - TODO 트리 조회 API 와 항목 생성/수정/삭제/완료 처리 API 를 추가한다.
   - 업무 상세 응답에 TODO 요약 집계를 포함할지 결정하고 반영한다.
   - 저장/수정/삭제 시 같은 업무 내 트리 재계산 로직을 서비스에 구현한다.
3. 프런트 타입과 API 연동을 추가한다.
   - TODO 응답/요청 타입을 `features/work/types` 와 `features/work/api` 에 추가한다.
   - 완료 숨기기, 접기/펼치기, 선택 업무 로딩 상태를 다루는 composable 을 만든다.
4. 신규 메뉴와 화면을 구현한다.
   - `navigation-menu.ts` 에 `업무 세부 작업` 메뉴를 추가한다.
   - 업무 선택 또는 상세 진입 라우트를 추가한다.
   - TODO 행에 체크박스, 제목, 진행률, 상태, 일정, 액션 버튼을 배치한다.
5. 검증과 회귀 확인을 진행한다.
   - TODO 계산 규칙 테스트를 작성한다.
   - 기존 업무관리 화면과 신규 상세 화면이 함께 동작하는지 확인한다.

## Done Criteria

- 한 업무에 여러 TODO 를 부모/자식 트리로 저장하고 조회할 수 있다.
- leaf TODO 는 직접 완료 체크가 가능하고, 상위 TODO 는 하위 완료율 기준으로 진행률과 상태가 자동 반영된다.
- 일정이 지난 미완료 항목은 지연 상태로 표시된다.
- 같은 부모 아래 정렬 순서가 유지된다.
- 화면에서 접기/펼치기, 완료 항목 숨기기, 하위 추가/수정/삭제가 가능하다.
- 신규 메뉴 `업무 세부 작업` 에서 업무별 TODO 상세를 확인하고 편집할 수 있다.
- 업무 자체 진행률을 TODO 완료율 기준으로 계산할 수 있는 API/모델 여지가 확보된다.
