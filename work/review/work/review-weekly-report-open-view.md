# 주간보고 공개 조회 기능 검토

## Scope

- 대상: 주간보고 메뉴를 `내 주간보고`, `인별 주간보고 조회`로 분리하고 공개된 주간보고를 타 사용자 기준으로 조회하는 기능
- 관점: 프론트 라우트 구조, 메뉴 진입점, 공개 주간보고 조회 API 필요 여부, 기존 주간보고 작성/수정 흐름과의 충돌 여부

## Summary

- 한줄 요약: 현재 구조에는 내 주간보고 관리 기능만 있고, 공개된 주간보고를 타 사용자 기준으로 조회하는 메뉴와 API가 모두 없다.
- 전체 판단: 프론트와 백엔드를 함께 확장해야 하므로 HEAVY 작업으로 분류한다.

## Findings or Scores

- 항목: 현재 주간보고 메뉴 구조
- 점수: 2/5
- 근거: `/work/weekly-reports` 단일 메뉴만 존재하고 화면도 `내 주간보고`만 렌더링한다.
- 영향: 사용자 관점에서 공개 여부를 설정해도 타인이 볼 수 있는 진입점이 없다.

- 항목: 공개 주간보고 조회 API
- 점수: 1/5
- 근거: `WeeklyReportController`와 `WeeklyReportService`는 로그인 사용자의 목록/상세만 제공한다.
- 영향: `인별 주간보고 조회`를 구현하려면 타 사용자 식별자와 공개 여부 조건을 반영한 목록/상세 API가 추가되어야 한다.

- 항목: 확장 난이도
- 점수: 3/5
- 근거: 기존 주간보고 VO/응답 DTO에 공개 여부와 게시 시각이 이미 존재한다.
- 영향: DB 구조 변경 없이 조회 조건과 화면 구성 추가 중심으로 구현 가능하다.

## Recommended Next Actions

1. 공개 주간보고 조회용 백엔드 API 범위를 정의한다.
2. 주간보고 화면을 `내 주간보고`와 `인별 주간보고 조회`로 분리한다.
3. 메뉴, API 연동, 공개 여부 조건을 함께 검증한다.
