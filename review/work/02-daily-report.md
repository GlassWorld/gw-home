## 작업 개요

사용자가 하루 단위로 자신의 업무 내용을 기록하고, 관리자는 전체 현황을 조회할 수 있는 일일보고 기능이다.
이번 설계의 핵심은 "개인 작성/개인 수정" 원칙을 지키면서도, 관리자 조회와 누락 관리, 내용 검색, 그리고 향후 주간보고 전환을 지원할 수 있는 구조를 먼저 잡는 것이다.

일일보고는 단순 메모가 아니라 날짜를 기준으로 한 업무 이력 데이터로 본다.
따라서 작성일자 중복 정책, 상태값 관리, 권한 경계, 누락 판단 기준을 처음부터 명확히 두는 편이 이후 운영 혼선을 줄인다.
특히 일일보고는 개인 전용 데이터이고, 주간보고는 오픈된 메뉴를 통해 타인과 공유되는 보고 기능이므로 두 기능은 같은 "보고" 범주라도 공개 범위와 책임 모델을 분리해서 봐야 한다.

## 핵심 설계 방향

### 1. 일일보고는 "회원 + 작성일자" 기준의 단일 문서로 본다

- 같은 사용자가 같은 날짜에 여러 건을 작성하게 두면 누락 판단과 주간 집계가 복잡해진다.
- 따라서 1차는 `회원 + 작성일자` 당 1건만 허용하고, 해당 보고 안에 업무내용/특이사항을 누적 서술하는 구조가 단순하다.
- 이 방식은 "그날 보고를 썼는가"를 빠르게 판별할 수 있어 누락 관리와 관리자 조회에 유리하다.

### 2. 작성일자와 생성시각을 분리한다

- 사용자가 기록하는 기준일은 `rpt_dt`이고, 실제 저장/수정 시각은 `created_at`, `updated_at`이다.
- 이렇게 분리해야 과거 날짜 보강 작성, 당일 수정, 관리자 감사 로그를 모두 자연스럽게 처리할 수 있다.

### 3. 상태는 보고 단위의 진행상태로 고정한다

- 요구사항상 상태는 `예정 / 진행중 / 완료`이며, 이번 범위에서는 보고 전체 상태로 관리하는 편이 단순하다.
- 업무별 세부 상태까지 쪼개면 별도 하위 테이블이 필요하므로 1차 범위를 넘어간다.
- 주간보고 확장 시에도 "해당 날짜 보고의 대표 상태"는 그대로 재사용 가치가 높다.

### 4. 누락 관리는 "별도 저장"보다 "조회 계산"을 우선한다

- 누락일을 테이블에 따로 저장하면 기준 변경 시 정합성이 깨지기 쉽다.
- 기본은 "조회 기간 내 근무대상일 - 작성 완료일"을 계산하는 방식이 안전하다.
- 필요 시 성능 문제를 보고 배치성 집계 테이블을 2차로 고려하는 것이 낫다.

### 5. 주간보고는 일일보고와 분리된 "공개 보고"로 본다

- 일일보고는 개인 회고와 누락 관리가 목적이므로 기본적으로 비공개가 맞다.
- 반면 주간보고는 오픈된 메뉴에 게시되면 타인과 공유되는 보고 기능이므로 공개 범위를 전제로 한 별도 모델이 필요하다.
- 따라서 "일일보고를 그대로 타인에게 노출"하는 방식보다, 일일보고를 참고해 주간보고를 작성하거나 발행하는 구조가 안전하다.

## 기능 요구사항 정리

### 1. 사용자 작성

- 사용자는 본인 계정 기준으로 일일보고를 작성할 수 있다.
- 필수 항목은 `작성일자`, `업무내용`, `진행상태`다.
- `특이사항`은 선택 입력으로 둔다.
- 동일 날짜에 이미 작성한 보고가 있으면 신규 생성 대신 수정 흐름으로 유도하는 편이 낫다.

이유:
날짜 단위 단건 정책과 맞물려 중복 생성을 막아야 누락 판정, 최신순 목록, 주간 집계가 모두 단순해진다.

### 2. 사용자 수정

- 본인이 작성한 보고만 수정 가능하다.
- 작성일자 수정 허용 여부는 신중히 봐야 한다.
- 1차는 `업무내용`, `진행상태`, `특이사항` 수정만 허용하고 `작성일자`는 생성 후 변경 불가가 안전하다.

이유:
작성일자까지 자유롭게 바뀌면 동일 날짜 중복, 누락 판정 왜곡, 관리자 감사 추적 난도가 올라간다.

### 3. 사용자 목록 조회

- 기본 목록은 본인 보고만 조회한다.
- 정렬은 `rpt_dt DESC`, 동률이면 `updated_at DESC`가 적절하다.
- 월 단위 또는 기간 단위 필터를 우선 제공하고, 내용 기반 검색은 보조 기능으로 둔다.

이유:
일일보고는 대부분 날짜 회고 중심으로 사용되므로 검색보다 기간 탐색이 우선 사용 흐름에 가깝다.

### 4. 관리자 전체 조회

- 관리자는 전체 회원의 일일보고를 조회할 수 있다.
- 조회 필터는 최소 `회원`, `작성일자 범위`, `진행상태`, `내용 키워드`, `작성 여부`가 필요하다.
- 관리자는 조회 권한만 우선 부여하고, 타인 수정은 1차 범위에서 제외하는 편이 안전하다.

이유:
운영상 필요한 것은 우선 "현황 파악"이지 "대리 수정"이 아니다.
수정 권한까지 열면 감사 책임과 권한 충돌이 커진다.

### 5. 공개 공유 범위

- 일일보고는 본인과 관리자만 접근 가능해야 한다.
- 타인 사용자는 일일보고를 조회할 수 없다.
- 타인과 공유되는 보고는 주간보고에서만 허용하는 것이 맞다.

이유:
일일보고와 주간보고의 공개 범위를 섞으면 개인 메모와 공유 보고의 성격이 충돌한다.

### 5. 내용 검색

- `업무내용`, `특이사항` 기준으로 검색한다.
- PostgreSQL에서는 1차는 `ILIKE` 기반 검색으로 시작하고, 데이터가 커지면 전문검색 확장을 검토한다.

이유:
요구사항 충족에는 부분 문자열 검색이면 충분하고, 초기 구현 비용도 낮다.

### 6. 누락 관리

- 특정 기간에 대해 "작성하지 않은 날짜"를 확인할 수 있어야 한다.
- 기본 기준은 평일 전체 또는 별도 설정된 근무일 캘린더와 비교하는 방식이 된다.
- 1차 구현에서는 "주말 제외 평일" 기준을 기본값으로 두고, 공휴일/휴가 반영은 후속 확장으로 분리하는 것이 현실적이다.

이유:
누락 관리는 기준일 정의가 핵심인데, 휴가/공휴일까지 한 번에 풀려면 추가 도메인이 필요하다.
먼저 단순한 기준으로 시작해야 기능 도입 속도와 운영 이해도가 높다.

## DB 구조 설계

### 테이블: `tb_wrk_dly_rpt`

```sql
CREATE TABLE tb_wrk_dly_rpt (
    wrk_dly_rpt_idx     BIGSERIAL    PRIMARY KEY,
    wrk_dly_rpt_uuid    UUID         NOT NULL UNIQUE DEFAULT gen_random_uuid(),
    mbr_acct_idx        BIGINT       NOT NULL,
    rpt_dt              DATE         NOT NULL,
    cntn                TEXT         NOT NULL,
    sts                 VARCHAR(20)  NOT NULL,
    note                TEXT,
    created_by          VARCHAR(100) NOT NULL,
    updated_by          VARCHAR(100),
    created_at          TIMESTAMPTZ  NOT NULL DEFAULT now(),
    updated_at          TIMESTAMPTZ  NOT NULL DEFAULT now(),
    del_at              TIMESTAMPTZ,
    CONSTRAINT uq_wrk_dly_rpt_owner_date UNIQUE (mbr_acct_idx, rpt_dt)
);

CREATE INDEX idx_wrk_dly_rpt_owner_date
    ON tb_wrk_dly_rpt (mbr_acct_idx, rpt_dt DESC);

CREATE INDEX idx_wrk_dly_rpt_date
    ON tb_wrk_dly_rpt (rpt_dt DESC);

CREATE INDEX idx_wrk_dly_rpt_status_date
    ON tb_wrk_dly_rpt (sts, rpt_dt DESC);
```

### 컬럼 메모

| 컬럼 | 설명 |
|------|------|
| `mbr_acct_idx` | 작성자 회원 PK 참조 |
| `rpt_dt` | 업무 기준일 |
| `cntn` | 업무내용 |
| `sts` | 진행상태 (`PLANNED`, `IN_PROGRESS`, `DONE`) |
| `note` | 특이사항 |

### 상태값 정의

| `sts` 값 | 표시명 |
|----------|--------|
| `PLANNED` | 예정 |
| `IN_PROGRESS` | 진행중 |
| `DONE` | 완료 |

### 검색 확장 인덱스 검토

- 1차는 일반 인덱스 + `ILIKE` 검색으로 시작한다.
- 데이터가 누적되면 `to_tsvector` 기반 GIN 인덱스 또는 trigram 인덱스를 추가 검토한다.

이유:
초기에는 운영 복잡도를 낮추고, 실제 검색 사용량이 확인된 뒤 최적화하는 편이 낫다.

## DB 설계 이유

### 왜 별도 상세 테이블을 두지 않는가

- 현재 요구사항은 단일 본문 + 상태 + 특이사항이면 충분하다.
- 상세 행 테이블까지 도입하면 API와 화면 모두 복잡해진다.
- 1차는 "하루 1문서" 모델이 요구사항에 가장 가깝다.

### 왜 주간보고 테이블과 분리해야 하는가

- 주간보고는 타인과 공유되는 공개 보고라서 권한, 조회 범위, 게시 상태 개념이 달라진다.
- 일일보고 테이블 하나로 개인 기록과 공개 게시를 함께 처리하면 권한 조건이 복잡해지고 실수 가능성이 커진다.
- 따라서 주간보고는 별도 테이블로 두고, 필요하면 일일보고를 참조해 내용을 구성하는 편이 안전하다.

### 왜 삭제보다 수정 중심으로 보는가

- 보고 데이터는 업무 이력 성격이 강하다.
- 물리 삭제나 잦은 삭제는 관리자 조회와 감사 추적을 약하게 만든다.
- 기본은 수정 위주로 두고, 삭제가 필요하면 소프트 삭제로만 제한하는 편이 안전하다.

## 추가 검토 요청 (2026-03-30)

기존 일일보고 기능 운영 중 확인된 수정/개선 요청을 아래 범위로 추가 검토한다.

### 수정 요청

1. 화면이 큰 상태에서 업무 선택 체크 시 버튼 또는 선택 UI가 비정상적으로 커지는 현상 수정
2. 일일보고 작성 시 `특이사항`이 본문 조합 또는 다른 영역 로직에 침범하는 현상 수정

### 개선 요청

1. 일일보고 본문에도 마크다운 형식 적용
2. 3분할 작성 레이아웃에서 좌/중 패널 폭은 고정하고, 우측 보고서 작성 영역은 더 넓게 사용

### 변경 목적

- 대형 화면에서 레이아웃이 흔들리지 않도록 선택 UI의 크기와 배치를 안정화한다.
- `오늘 수행 내용`과 `특이사항`의 의미와 저장 범위를 다시 분리해 작성 의도를 보존한다.
- 기존 Vault 메모와 유사하게 일일보고도 가독성 있는 마크다운 작성 경험을 제공한다.
- 작성 영역을 넓혀 실제 보고 입력에 집중할 수 있는 편집 흐름으로 보정한다.

### 예상 영향 범위

| 구분 | 변경 내용 |
|------|-----------|
| Frontend - work | `pages/work/daily-reports/index.vue`, `ReportEditorPanel.vue`, `TaskListPanel.vue` 레이아웃/선택 UI 수정 |
| Frontend - work | 일일보고 마크다운 preview 또는 렌더링 처리 추가 |
| Frontend - types/composable | 저장 payload 조합 규칙이 `특이사항` 분리 정책과 맞는지 재점검 |
| Backend - work | 현재 저장/조회 시 `cntn`, `note` 매핑이 섞이는지 확인 필요 |

| 프론트에서만 본문/특이사항을 분리하고 백엔드 조합 규칙이 그대로 남아 데이터가 다시 섞일 수 있음 | 저장 payload와 조회 응답 조합 규칙을 함께 점검 |

### 작업 분류 재판단

- 현재 추가 요청은 기존 일일보고 기능의 **Frontend 중심 개선 + 저장 정합성 점검** 성격이 강하다.
- 다만 `특이사항` 침범 원인이 API 조합 규칙에 있으면 Backend 동시 수정이 필요하므로, 실행 단계에서는 **Full-stack HEAVY 가능성**을 다시 확인해야 한다.

## 추가 검토 메모 (2026-03-29, 3분할 작성 모달 개편안)

### 검토 요청 요약

- 일일보고 작성 모달을 좌/중/우 3분할 구조로 개편하고 싶다.
- 좌측은 업무 목록, 중앙은 선택 업무 기준 지난 일일보고 내역, 우측은 실제 작성 폼으로 구성하고 싶다.
- 컴포넌트는 `TaskListPanel`, `ReportHistoryPanel`, `ReportEditorPanel`로 분리하고 싶다.
- 더미 데이터 기반으로 먼저 동작 가능한 형태를 원하지만, 실제 API 연동 구조까지 고려되어야 한다.
- 입력 피로도를 줄이는 UX와 이전 보고 기반 자동완성 아이디어도 함께 검토가 필요하다.

### 검토 결과

#### 1. 3분할 레이아웃 자체는 현재 방향과 잘 맞는다

- 현재 일일보고는 이미 "업무등록 선택 + 보고 작성" 방향으로 이동해 있다.
- 따라서 좌측에 업무 목록, 우측에 작성 폼을 두는 것은 기존 의도와 자연스럽게 연결된다.
- 특히 데스크탑 기준에서 업무 선택과 작성 맥락을 한 화면에 유지하는 것은 사용 흐름 개선 효과가 크다.

정리:
레이아웃 방향은 타당하고, 단순 UI 개편 수준이 아니라 "선택 업무 중심 작성 경험"을 강화하는 작업으로 보는 것이 맞다.

#### 2. 중앙 "선택 업무 기준 과거 보고 내역"은 신규 API 또는 조회 조건 확장이 필요하다

- 현재 일일보고 API는 기간/키워드 기준 목록 조회만 제공한다.
- "선택된 업무 1건을 기준으로 최근 보고만 별도 조회"하는 전용 응답은 아직 없다.
- 지금 구조는 일일보고 1건에 여러 업무가 연결되므로, 중앙 패널은 사실상 "이 업무가 포함된 최근 일일보고" 목록을 의미하게 된다.

즉 필요한 해석:

- `selectedTask = work_unit_uuid`
- `reportList = 해당 work_unit_uuid 가 연결된 daily report 목록`

필요한 백엔드 변화:

- `GET /api/v1/daily-reports/history?workUnitUuid=...`
- 또는 기존 목록 API에 `workUnitUuid` 필터 추가

이유:
중앙 패널은 작성 중 참고용으로 자주 호출될 가능성이 높아, 전체 목록을 프론트에서 재가공하는 방식보다 전용 필터 API가 더 단순하고 안정적이다.

#### 3. 우측 작성 폼의 `제목 / 오늘 수행 내용 / 이슈 / 다음 계획`은 현재 저장 모델과 1:1로 맞지 않는다

- 현재 일일보고 저장 모델은 `선택된 업무 목록 + 특이사항(note)`에 가깝다.
- 제안된 우측 폼은 사실상 구조화된 보고 본문 모델이다.
- 따라서 이 필드를 그대로 도입하면 최소한 아래 둘 중 하나를 정해야 한다.

선택지 A:

- UI에서만 필드를 나누고 저장 시 하나의 조합된 본문 또는 note 로 합친다.
- 구현은 빠르지만 검색/자동완성/주간보고 재사용성이 떨어진다.

선택지 B:

- 일일보고에 `title`, `workSummary`, `issue`, `nextPlan` 같은 구조화 필드를 추가한다.
- 후속 활용성이 좋지만 DB/API 스펙 변경이 다시 발생한다.

권고:
이번 1차에서는 우측 폼을 구조화 UI로 두더라도 저장은 `note` 조합 방식으로 시작하고, 실제 사용성이 검증되면 구조화 필드 분리 여부를 2차로 판단하는 편이 안전하다.

#### 4. "선택된 업무 자동 바인딩"은 단일 선택 기준으로 명확히 해야 한다

- 현재 일일보고는 여러 업무를 한 보고에 연결할 수 있다.
- 그런데 중앙 패널은 "선택된 업무 기준 과거 보고"를 전제로 하므로, UX 중심축은 사실상 단일 활성 업무가 된다.
- 따라서 상태는 `선택된 업무(active) 1개`와 `최종 저장될 연결 업무 목록(selected) N개`를 분리하는 편이 좋다.

예시:

- `activeTaskUuid`: 중앙 패널 조회 기준
- `selectedTaskUuids`: 실제 저장 대상 업무 목록

이 분리를 하지 않으면:

- 중앙 패널이 어떤 업무 기준으로 바뀌는지 모호해지고
- 다중 선택과 히스토리 패널의 관계가 헷갈릴 수 있다.

#### 5. 컴포넌트 분리는 적절하지만, 상태 소유는 페이지가 가져야 한다

- `TaskListPanel`, `ReportHistoryPanel`, `ReportEditorPanel` 분리는 화면 복잡도를 줄이는 데 유리하다.
- 다만 상태는 각 패널이 따로 소유하기보다 상위 페이지가 일원화해 전달하는 편이 맞다.

권장 상태 구조:

- `activeTask`
- `selectedTaskUuids`
- `reportHistory`
- `historyPreviewReport`
- `reportForm`
- `isTaskLoading`
- `isHistoryLoading`
- `isSubmitting`

이유:
패널 간 상호작용이 강하므로 로컬 컴포넌트 상태로 쪼개면 동기화 비용이 커진다.

#### 6. skeleton UI 와 empty state 는 꼭 필요하지만, "히스토리 패널" 우선 적용이 체감효과가 가장 크다

- 좌측 업무 목록은 이미 사용자가 리스트 로딩에 익숙한 영역이라 일반 로딩 문구로도 버틸 수 있다.
- 반면 중앙 히스토리 패널은 "선택 후 반응성"이 핵심이라 비어 보이면 끊긴 느낌이 강하다.
- 따라서 skeleton 우선순위는 중앙 > 좌측 > 우측이 적절하다.

권장 empty state 문구:

- 업무 미선택: `업무를 선택하면 이전 일일보고를 확인할 수 있습니다`
- 이력 없음: `이 업무에 대한 이전 보고가 없습니다`
- 검색 결과 없음: `조건에 맞는 업무가 없습니다`

- 업무 선택 시 우측 폼 상단에 선택 업무명을 자동 주입해 현재 맥락을 고정한다.
- 이전 보고에서 자주 쓰는 문구를 `칩 버튼` 형태로 노출해 한 번 클릭으로 삽입할 수 있게 한다.
- `오늘 수행 내용`, `이슈`, `다음 계획`은 placeholder 를 설명형이 아니라 예시형으로 제공한다.
- `선택 업무 기준 최근 작성 패턴`을 우측에 짧은 힌트로 보여줘 첫 문장 작성 부담을 줄인다.
- `저장` 외에 `이전 보고 기반 초안 넣기` 버튼을 제공하면 빈 화면 공포를 줄일 수 있다.

### 이전 보고 기반 자동완성 제안

#### 1. 템플릿 자동완성

- 최근 동일 업무 보고에서 `오늘 수행 내용 / 이슈 / 다음 계획`을 분리 추정해 초안으로 채운다.
- 사용자는 그대로 수정만 하면 되므로 입력 부담이 크게 줄어든다.

#### 2. 문구 추천 자동완성

- 동일 업무의 최근 5건에서 자주 나온 시작 문장/핵심 문구를 추천한다.
- 예:
  - `API 응답 구조 정리 및 예외 케이스 확인`
  - `화면 연동 후 저장/조회 흐름 점검`

#### 3. 다음 계획 연속 자동완성

- 전일 보고의 `다음 계획`을 오늘 보고의 `오늘 수행 내용` 초안 후보로 제안한다.
- 이 흐름은 실제 업무 연속성과 맞아서 체감 효율이 높다.

#### 4. 이슈 재노출 제안

- 최근 보고에 미해결성 표현이 있으면 `이번 보고에도 이어서 기록할까요?` 형태로 제안한다.
- 예:
  - `배포 후 인증 검증 필요`
  - `예외 케이스 추가 확인 필요`

### 최종 판단

- 3분할 모달 개편안은 UX 측면에서 충분히 가치가 있다.
- 다만 중앙 히스토리 패널과 우측 구조화 입력은 현재 API/저장모델과 완전히 일치하지 않으므로, UI 개편과 API 확장을 단계적으로 나누는 편이 안전하다.
- 가장 현실적인 진행 순서는 `3분할 UI + 단일 activeTask 상태 + 업무별 히스토리 API 추가 검토 + 저장은 기존 모델 호환` 순서다.

판단:
- 이 요구를 화면 해석 수준으로만 반영하면 프론트 수정으로 가능하다.
- 하지만 "업무등록의 업무만 등록 가능"을 데이터적으로 보장하려면 일일보고와 업무등록 사이 연결 모델이 추가로 필요하다.
- 따라서 요구 의미를 먼저 확정해야 한다.

#### 2. 진행상태 제거는 작성 모달만 지우는 수준인지, 일일보고 자체 모델에서 없애는지 구분이 필요하다

- 현재 `sts`는 DDL, VO, DTO, 서비스 검증, 사용자 목록, 관리자 조회, 주간보고 참고 데이터, 주간보고 AI 초안 생성까지 연결되어 있다.
- 그래서 "작성 화면에서만 숨김"은 프론트 중심 변경으로 끝날 수 있지만,
- "일일보고에서 상태 개념 자체 제거"는 DB, API, 관리자 화면, 주간보고까지 함께 손봐야 한다.

판단:
- 요청 문맥상 "일일보고 작성에서 진행상태는 무의미"라는 표현이므로 1차는 작성/수정 UI에서 제거하고, 백엔드는 기본값 유지 또는 숨김 처리로 정리하는 접근이 가장 안전하다.
- 상태 컬럼 자체 삭제는 별도 HEAVY 작업으로 분리하는 편이 맞다.

#### 3. 업무 번호 구분 제거는 타당하다

- 현재 화면은 `workItems` 배열을 사용해 여러 개의 textarea를 만들고, 저장 시 빈 줄 두 개로 합쳐 `cntn`에 저장한다.
- 이 방식은 "업무등록 연계"가 아니라 "본문 분할 편집"에 가깝고, 사용자가 보기에는 업무가 개별 엔터티처럼 오해될 수 있다.
- 특히 `업무 1`, `업무 2` 라벨은 현재 데이터 모델보다 더 강한 구조를 암시한다.

판단:
- 번호형 반복 입력 UI는 제거하는 쪽이 요구와 더 잘 맞는다.
- 다만 자유 입력 단일 textarea로 바꿀지, 업무등록 옵션을 선택해 누적하는 UI로 바꿀지는 별도 결정이 필요하다.

#### 4. 모달 확대는 단독으로 위험도가 낮다

- 현재 모달 너비는 `1120px`로 이미 비교적 큰 편이지만, 좌측 작성 폼과 우측 최근 보고 패널을 함께 보여줘 실제 체감 공간은 좁을 수 있다.
- 번호형 입력 UI가 단일 입력 UI로 바뀌면 같은 폭에서도 훨씬 여유로워질 가능성이 있다.

판단:
- 모달 크기 확대 자체는 LOW 리스크다.
- 다만 본질적인 답답함이 레이아웃 구조에서 오므로, 폭 조정은 입력 방식 정리와 같이 보는 편이 효과적이다.

#### 5. 목록 기본 조회 기간은 "월초 ~ 오늘"이 아니라 "최근 2주 ~ 오늘"으로 바뀌어야 한다

- 현재 프론트 기본 필터는 월초부터 오늘까지로 초기화된다.
- 현재 백엔드 기본 조회값도 요청 파라미터가 없으면 월초부터 오늘까지로 계산한다.
- 따라서 화면 초기값만 바꾸면 기본 API 호출 결과와 불일치할 수 있고, 반대로 백엔드만 바꾸면 화면 초기 표시와 어긋날 수 있다.

판단:
- 이 요구는 프론트/백엔드 둘 다 같은 기준으로 맞춰야 한다.
- 기본 조회 기간은 "오늘 포함 최근 14일"인지, "오늘 포함 2주 캘린더 범위"인지 표현을 명확히 해야 하지만, 현재 문맥상 `오늘 기준 최근 2주`를 `오늘 ~ 14일 전`의 rolling window로 해석하는 것이 자연스럽다.
- 예를 들어 현재 날짜가 `2026-03-29`라면 기본 조회 범위는 `2026-03-15`부터 `2026-03-29`까지로 보는 해석이 가장 단순하다.

### 권장 구현 방향

#### 방향 A. 범위 최소화

- 일일보고는 계속 단일 본문(`cntn`)으로 유지
- 작성/수정 모달에서 진행상태 입력 제거
- 업무 `1, 2, 3` 반복 UI를 단일 textarea 또는 단순 줄바꿈 입력으로 변경
- 업무등록은 "참고/재사용 가능한 템플릿" 정도로만 활용
- 목록 기본 조회 기간을 `월초 ~ 오늘`에서 `최근 2주 ~ 오늘`으로 변경

장점:
- 기존 DB/API 영향이 작다.
- 비교적 빠르게 요구에 맞는 UX 정리가 가능하다.

한계:
- "일일보고에 등록하는 업무는 업무등록의 업무"를 구조적으로 보장하지 못한다.

#### 방향 B. 의미를 엄격히 반영

- 일일보고 작성 시 업무등록 목록에서 업무를 선택해 추가하는 방식으로 변경
- 필요 시 일일보고 상세 항목 테이블 또는 연결 테이블을 새로 설계
- 주간보고가 참조하는 원본 데이터도 연결 구조에 맞게 재정의

장점:
- 요구 의미를 가장 정확히 반영할 수 있다.

한계:
- DB/API/프론트 전부 수정하는 HEAVY 작업이 된다.
- 주간보고, 관리자 조회, 검색 방식도 함께 재설계해야 할 수 있다.

### 현재 기준 권고

- 이번 요청은 우선 방향 A로 정리하는 것이 현실적이다.
- 즉, "번호형 업무 입력 제거 + 작성 화면 상태 제거 + 모달 여백 확대"를 먼저 적용하고,
- 그리고 기본 조회 기간을 `최근 2주`로 함께 정리하고,
- "업무등록 업무만 선택 가능"까지 강제할지는 별도 확정 후 다음 단계에서 진행하는 편이 안전하다.

### 계획 전 확인할 쟁점

- 일일보고 본문은 자유 입력을 유지해도 되는지
- 아니면 업무등록의 업무 선택만 허용해야 하는지
- 진행상태는 "작성 화면에서만 숨김"인지, "일일보고 도메인에서 완전 제거"인지
- 최근 작성 보고 패널은 유지할지, 업무등록 선택 패널로 대체할지
- 최근 2주 범위는 `오늘 포함 14일`로 볼지, 정확히 `지난 2주간 평일 기준`으로 볼지

### 왜 `DATE` 타입을 쓰는가

- 작성일자는 시간대보다 "캘린더 날짜"가 본질이다.
- `TIMESTAMPTZ`로 저장하면 날짜 경계 해석이 오히려 복잡해질 수 있다.
- 반면 생성/수정 시각은 감사 목적이라 `TIMESTAMPTZ`가 적절하다.

## API 설계 이유

### 왜 사용자/관리자 엔드포인트를 분리하는가

- 같은 테이블을 보더라도 조회 관점과 필터가 다르다.
- 사용자 API는 본인 데이터만 단순하게 다루고, 관리자 API는 회원 필터/누락 현황 같은 운영 기능에 집중하는 편이 명확하다.
- 권한 분기도 URL 레벨에서 드러나 유지보수가 쉽다.

### 왜 수정은 `PUT` 하나로 제한하는가

- 상태, 내용, 특이사항이 하나의 문서 단위로 묶여 있어 부분 업데이트보다 전체 수정이 단순하다.
- 필드가 적은 1차 설계에서는 `PATCH`보다 `PUT`이 운영 이해도가 높다.

### 왜 누락 조회를 별도 API로 분리하는가

- 누락 관리는 일반 목록과 계산 로직이 다르다.
- 목록 응답에 항상 누락 정보를 섞으면 쿼리 복잡도와 응답 비용이 커진다.
- 별도 API로 분리해야 관리자 대시보드, 개인 알림, 월간 점검에 재사용하기 쉽다.

## 권한 정책

### 사용자

- 본인 보고만 작성 가능
- 본인 보고만 조회 가능
- 본인 보고만 수정 가능
- 타인 보고 UUID를 알아도 접근 불가

이유:
보고 내용은 개인 업무 이력이라 기본 보호 범위를 강하게 잡아야 한다.

### 관리자

- 전체 보고 목록/상세 조회 가능
- 전체 누락 현황 조회 가능
- 1차 범위에서는 타인 보고 수정/삭제 권한은 부여하지 않음

이유:
운영 관리자에게 필요한 것은 우선 모니터링과 점검이다.
편집 권한까지 부여하면 데이터 책임 주체가 흐려진다.

### 시스템 정책

- 소유권 판단 기준은 로그인 계정의 `mbr_acct_idx`다.
- 조회/수정 SQL 모두 `mbr_acct_idx` 조건을 강제해야 한다.
- 관리자 조회만 예외적으로 전체 범위를 허용한다.

이유:
컨트롤러 레벨 체크만으로는 누락되기 쉬우므로 서비스와 SQL 양쪽에서 소유권 조건이 명확해야 한다.

## 확장 가능성

### 1. 주간보고

- 일일보고를 참고해 주간 단위 보고를 작성할 수 있다
- 주간보고는 오픈 메뉴 게시와 공유 범위를 가져야 하므로 별도 테이블과 게시 상태가 필요하다
- 일일보고는 주간보고 작성을 위한 개인 원본 데이터로 활용될 수 있다
- 주간보고 작성 방식은 `AI 보조 작성`과 `수동 작성`을 함께 지원하는 편이 적절하다

이유:
개인 기록과 공개 보고를 분리해야 보안과 UX가 모두 단순해진다.

### 2. 통계

- 개인별 작성률, 상태 분포, 누락률, 최근 작성 추세를 계산할 수 있다
- 관리자 대시보드나 팀 운영 지표로 확장 가능하다

이유:
날짜와 상태가 구조화돼 있으면 통계는 추가 조회 계층만으로 확장 가능하다.

### 3. 업무등록 연계

- 기존 `work unit`과 연결하려면 `daily_report` 하위 상세 테이블 또는 참조 컬럼이 추가될 수 있다
- 다만 1차는 자유 텍스트 중심이므로 연계를 강제하지 않는 편이 도입 장벽이 낮다

이유:
초기에는 입력 속도와 채택률이 중요하고, 구조화는 이후 점진적으로 강화하는 편이 낫다.

### 4. AI 요약

- 일/주/월 단위 업무 요약, 누락 경고, 반복 업무 추출에 활용 가능하다
- 다만 AI는 보조 수단으로 두고, 최종 주간보고 내용은 사용자가 직접 수정하고 확정할 수 있어야 한다

이유:
텍스트 본문과 상태, 날짜가 함께 쌓이면 AI 요약 품질이 높아진다.

### 주간보고 분리 설계 메모

- 후보 테이블: `tb_wrk_wkl_rpt`
- 핵심 필드 예시: `ttl`, `cntn`, `wk_start_dt`, `wk_end_dt`, `opn_yn`, `pbls_at`
- 공개 전에는 임시저장, 오픈 메뉴 등록 시에는 공유 상태로 전환하는 구조가 적절하다
- 주간보고 본문 생성 방식은 `manual`, `ai_assisted` 정도의 생성 유형을 남길 수 있다

### 주간보고 작성 화면 메모

- 화면은 `주간보고 편집 영역`과 `해당 주 일일보고 참고 영역`의 2패널 구성이 적절하다
- 한쪽 패널에는 주간보고 제목/본문/공개 여부/임시저장 및 게시 액션을 두고, 다른 패널에는 그 주의 일일보고 목록을 날짜순으로 보여준다
- 사용자는 일일보고를 보면서 직접 주간보고를 수동 작성할 수 있어야 한다
- AI 작성 버튼을 누르면 같은 주의 일일보고 묶음을 바탕으로 초안을 생성하되, 결과는 편집 가능한 초안으로만 넣는 것이 안전하다
- 일일보고 참고 패널에서는 날짜, 상태, 업무내용, 특이사항을 빠르게 훑을 수 있어야 하고, 특정 일자의 내용을 펼쳐 상세 확인할 수 있으면 좋다

이유:
주간보고는 공유되는 산출물이므로 AI 자동 게시보다 "참고자료 기반의 사용자 확정" 흐름이 더 안전하다.

이유:
주간보고는 "작성"뿐 아니라 "게시" 개념이 함께 있어 일일보고보다 상태 모델이 더 필요하다.

| AI가 생성한 주간보고 초안을 그대로 게시하면 부정확하거나 과한 표현이 노출될 수 있음 | 중간 | AI 결과는 초안으로만 두고, 사용자의 검토/수정 후 게시 |

## 권장 구현 순서

1. `tb_wrk_dly_rpt` DDL과 유니크 정책 확정
2. 사용자 작성/수정/목록 API 우선 구현
3. 관리자 전체 조회 API 추가
4. 누락 조회 API를 기간 계산 방식으로 구현
5. 이후 필요 시 주간보고 작성 화면, AI 보조 초안 기능, 발행 기능, 통계, 업무등록 연계 순서로 확장

이 순서가 적절한 이유는, 일일보고의 본질인 "개인 작성 데이터 축적"이 먼저 안정돼야 관리자 기능과 확장 기능도 의미를 갖기 때문이다.

## 결론

일일보고는 `work` 도메인 안에서 "회원 + 날짜" 기반의 단일 보고 문서로 설계하는 것이 가장 단순하고 안정적이다.
이 구조는 본인 작성/수정 권한을 명확히 지키면서, 관리자 전체 조회와 누락 관리, 그리고 공개형 주간보고로의 분리 확장까지 자연스럽게 이어진다.

핵심 결정은 세 가지다.

- 같은 회원의 같은 날짜 보고는 1건으로 제한한다.
- 누락 관리는 저장이 아니라 조회 계산으로 처리한다.
- 관리자 권한은 1차에서 조회 중심으로 제한한다.
- 주간보고는 일일보고와 분리된 공개 보고 모델로 설계한다.

이 세 가지를 먼저 고정하면 이후 기능 확장 시에도 구조가 크게 흔들리지 않는다.
