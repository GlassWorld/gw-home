# Review: 전체 리팩토링 점검

> 작업 레벨: **HEAVY** (문서 + DDL + backend + frontend 전반 점검)
> 관련 review: `review/work/01-work-unit-management.md`, `review/work/02-daily-report.md`
> 관련 todo: `todo/work/03-overall-refactor-batch.md`

---

## 검토 개요

문서, DDL, 소스코드, Mapper, 프론트 API 연동부를 기준으로 리팩토링 이후 전반 정합성을 점검했다.
이번 검토의 목적은 "규칙 위반 여부"보다도 "리팩토링 후 문서와 실제 구현이 서로 맞는지", "겉으로는 동작하지만 운영 중 틀어질 지점이 있는지"를 찾는 데 있다.

점검 범위는 다음 네 축으로 제한했다.

- 문서: 인증 흐름, API 계약, 페이지 구조, DB 문서
- DDL: 개별 도메인 DDL과 집계 DDL 문서 정합성
- Backend: Controller, Service, DTO, Mapper XML
- Frontend: composable, 타입 정의, 페이지 라우팅

---

## 주요 점검 결과

### 1. 인증 문서와 실제 구현이 크게 어긋나 있다

문서에는 리프레시 토큰을 `HttpOnly` 쿠키로 관리한다고 적혀 있다.
하지만 실제 구현은 로그인/토큰 갱신 응답에서 `refreshToken`을 바디로 내려주고, 프론트에서 이를 읽어 일반 쿠키에 저장하는 구조다.

이 구조는 다음 문제를 만든다.

- 문서 기준으로는 서버 주도 쿠키 인증처럼 보이지만 실제로는 프론트가 토큰 수명 관리 책임을 갖는다.
- JavaScript에서 리프레시 토큰 접근이 가능해져 XSS 발생 시 장기 세션 탈취 위험이 커진다.
- 운영자나 후속 개발자가 문서만 보고 보안 가정을 하면 잘못된 판단을 하게 된다.

정리하면 현재 구현은 "동작은 할 수 있지만, 문서와 보안 모델이 다르다"는 점이 가장 큰 문제다.

### 2. 관리자 일일보고 목록은 일부 데이터가 비어 보일 가능성이 있다

관리자 목록 조회 쿼리는 `spcl_note`만 조회하고, 본문 컬럼 `cntn`은 조회하지 않는다.
그런데 서비스 계층에서는 응답 생성 시 `cntn`을 그대로 사용하고, `note`가 비어 있으면 `cntn`으로 대체하도록 작성되어 있다.

이 조합은 다음 상황에서 바로 문제가 된다.

- 특이사항 없이 작성된 일일보고는 관리자 화면에서 내용이 비어 보일 수 있다.
- 프론트는 `note`만 표에 표시하므로, 실제 사용자는 "작성은 했는데 관리자 화면엔 안 보인다"고 느낄 수 있다.

즉 쿼리 투영 컬럼과 응답 조립 로직이 리팩토링 중 분리된 채 남아 있다.

### 3. 업무 정렬 기준은 스키마만 있고 실제 갱신 로직이 없다

`tb_work_unit`에는 `use_cnt`, `last_used_at` 컬럼이 있고, 업무 목록 정렬도 `recent`, `frequent` 옵션을 제공한다.
하지만 일일보고 저장 시 실제로 하는 일은 연결 테이블을 삭제 후 다시 저장하는 것뿐이며, 업무 사용 횟수나 최근 사용 시각은 갱신하지 않는다.

그 결과는 다음과 같다.

- "최근 사용순"이 실제 최근 사용을 반영하지 못한다.
- "자주 사용순"이 누적 사용량을 반영하지 못한다.
- UI에서 제공하는 정렬 옵션과 사용자가 체감하는 정렬 결과가 어긋난다.

초기 설계 문서에서는 해당 컬럼을 "추후 확장용"으로 두는 방향이었는데, 현재는 정렬 옵션이 먼저 노출되어 오해를 만든 상태다.

### 4. 문서와 집계 DDL이 현재 코드베이스를 따라오지 못했다

`docs/backend/database.md`는 `all-ddl.sql`이 전체 스키마 재생성 파일이라고 설명한다.
하지만 실제 `docs/all-ddl.sql`에는 `work` 관련 테이블이 전혀 반영되어 있지 않다.

또한 프론트 문서와 실제 페이지 구조도 일부 다르다.

- 문서에는 `/profile`, `/admin` 페이지가 있다고 적혀 있지만 실제 파일은 없다.
- 루트 페이지는 `/dashboard`로 즉시 이동하지만, 프로젝트 규칙과 문서 일부는 `/login`을 초기 진입점으로 설명한다.
- API 계약 문서는 `ApiResponse<T>`의 `data`를 nullable로 설명하지만, 프론트 공통 타입은 `data: T`로 고정되어 있다.
- 문서 예시의 페이지 응답은 `page: 0` 기반인데, 실제 백엔드는 1페이지부터 정규화한다.

이 상태에서는 문서가 구현 가이드 역할을 하지 못하고, 오히려 리팩토링 회귀를 숨길 가능성이 있다.

### 5. 서비스 내부 변환 메서드가 누적되어 책임이 무거워졌다

여러 서비스가 조회, 저장, 권한 검증뿐 아니라 DTO 변환까지 함께 들고 있다.
특히 `toResponse`, `toDetailResponse`, `toSummaryResponse` 계열 private 메서드가 서비스마다 반복되고 있어, 리팩토링 시 수정 포인트가 넓고 응답 조립 규칙도 도메인별로 흩어져 있다.

이 상태는 다음 문제를 만든다.

- 서비스가 비즈니스 로직보다 변환 코드까지 함께 품어 가독성이 떨어진다.
- 유사한 변환 로직이 각 서비스에 중복되어 수정 누락 위험이 커진다.
- 관리자 응답과 일반 응답처럼 파생 DTO가 늘어날수록 서비스 클래스가 빠르게 비대해진다.

특히 `work` 도메인은 일일보고, 주간보고, 업무등록이 함께 있어 변환 책임 분리 효과가 큰 편이다.

### 6. 기본 검증 함수가 서비스별로 반복되어 공통화 여지가 있다

`normalizeText`, `require*`, `validate*` 같은 기본 검증/정규화 함수가 서비스 내부에 반복된다.
물론 모든 검증을 공통화하면 안 되지만, 도메인 의미가 없는 수준의 기본 함수는 `share`로 정리할 여지가 보인다.

예를 들면 다음 항목이 후보가 될 수 있다.

- 공백 제거 후 빈 문자열을 null로 바꾸는 정규화
- 필수 문자열/날짜 값 확인
- 시작일과 종료일의 순서 검증
- 공통 enum 또는 상태값 후보 검증

반대로 중복 검사, 본인 소유 검증, 주간보고 7일 제한 같은 정책성 검증은 도메인 서비스에 남기는 편이 안전하다.

추가로 날짜 처리, 기본 형변환, null 체크 같은 범용 로직도 `share.util` 계층으로 올릴 여지가 있다.
이런 항목은 특정 도메인에 묶이지 않고 여러 서비스에서 반복되기 쉬워, 공통 유틸로 정리하면 서비스 코드가 훨씬 단순해질 수 있다.

후보 예시는 다음과 같다.

- 날짜 범위 비교
- `String -> LocalDate` 같은 기본 형변환 보조
- null 안전 처리
- 빈 문자열/공백 문자열 정리

다만 이 역시 "범용 도구"까지만 허용하고, 업무 정책 판단은 서비스에 남겨야 한다.

### 7. 서비스 주석과 로그 일관성도 함께 정리할 필요가 있다

프로젝트 규칙상 서비스 메서드에는 한글 주석과 진입/완료/실패 로그가 중요하다.
하지만 현재 코드는 도메인별 편차가 있어, 어떤 서비스는 로그가 비교적 잘 들어가 있고 어떤 서비스는 변환/검증 리팩토링을 거치면서 설명과 로그 밀도가 다시 들쭉날쭉해질 여지가 있다.

이번 구조 정리는 단순히 `convert` 분리와 `validate` 공통화만이 아니라, 서비스 메서드의 한글 주석과 로그 기준까지 함께 맞춰야 효과가 크다.
특히 리팩토링 이후에는 "무엇을 공통화했는지", "어디서 검증이 실패했는지"가 로그만으로 보이도록 정리하는 편이 운영과 유지보수에 유리하다.

### 8. Mapper alias도 자동 매핑 설정 기준으로 정리할 필요가 있다

현재 MyBatis 설정에는 `mapUnderscoreToCamelCase(true)`가 켜져 있다.
따라서 `use_yn AS useYn`, `use_cnt AS useCnt`, `last_used_at AS lastUsedAt`, `created_by AS createdBy`, `updated_by AS updatedBy`, `created_at AS createdAt`, `updated_at AS updatedAt` 같은 alias는 대부분 없어도 자동 매핑된다.

반면 `BaseVo`의 `idx`, `uuid`처럼 컬럼명과 필드명이 구조적으로 다른 경우는 alias가 여전히 필요하다.

즉 alias 정리는 다음 기준으로 보는 편이 맞다.

- 유지 필요:
  - `work_unit_idx AS idx`
  - `work_unit_uuid AS uuid`
  - `*_idx AS idx`
  - `*_uuid AS uuid`
- 제거 후보:
  - snake_case와 camelCase가 자동 변환으로 그대로 매핑되는 컬럼 alias

이 부분은 기능 버그보다는 Mapper 가독성과 유지보수성 문제에 가깝지만, 리팩토링 시 같이 정리하면 중복 alias를 많이 줄일 수 있다.

## 세부 메모

### 문서 측면

- 인증 흐름 문서는 현재 구현과 맞게 갱신하거나, 반대로 구현을 문서 기준으로 되돌리는 선택이 필요하다.
- 페이지 문서는 실제 존재하는 경로 기준으로 다시 정리해야 한다.
- API 계약 문서는 `null` 허용 여부와 페이지 시작 번호를 실제 백엔드/프론트 합의 기준으로 통일해야 한다.

### Backend 측면

- 관리자 일일보고 목록 쿼리에서 `cntn` 누락 여부를 우선 확인해야 한다.
- 응답 DTO 조립 시 조회 컬럼과 fallback 로직이 맞는지 다시 대조할 필요가 있다.
- 업무 사용 이력 컬럼을 살릴지, 아니면 정렬 옵션을 숨길지 방향 결정을 먼저 해야 한다.
- 서비스 내부 `toResponse` 계열 메서드는 도메인별 `convert` 클래스로 분리하는 편이 이후 유지보수에 유리하다.
- `normalize`/`validate` 계열 기본 함수는 도메인 중립 항목에 한해 `share` 공통화 후보로 볼 수 있다.
- 날짜 처리, 기본 형변환, null 체크 같은 범용 로직은 `share.util` 공통 유틸 후보로 볼 수 있다.
- 서비스 메서드 상단 한글 주석과 진입/완료/실패 로그도 이번 리팩토링 묶음에서 같이 정리하는 편이 안전하다.
- Mapper XML의 snake_case -> camelCase 자동 매핑 대상 alias는 제거하고, `idx`/`uuid`처럼 실제 alias가 필요한 컬럼만 남기는 정리도 가능하다.

### Frontend 측면

- `useAuth`는 현재 구조상 리프레시 토큰을 직접 다루므로, 보안 모델을 명확히 고정해야 한다.
- 공통 `ApiResponse` 타입은 실제 서버 계약을 반영하도록 다시 정의하는 편이 안전하다.
- 루트 라우팅(`/`) 정책은 문서와 구현 중 하나로 통일해야 한다.

### DDL 측면

- `work` 도메인 DDL이 개별 파일에는 존재하지만 집계 DDL에는 누락되어 있다.
- 운영/로컬 초기화 기준으로 `all-ddl.sql`을 계속 쓸 계획이라면 반드시 최신 상태로 맞춰야 한다.

## 결론

이번 리팩토링은 큰 방향 자체는 정리되어 있지만, 문서와 실제 구현이 서로 다른 가정을 가진 부분이 남아 있다.
특히 인증, 관리자 일일보고 조회, 업무 정렬 기준, 집계 DDL 문서가 대표적인 정합성 깨짐 지점이다.

추가로 서비스 내부 변환 메서드와 기본 검증 함수가 누적되면서 구조적 리팩토링 필요성도 드러났다.
다음 단계는 기능 추가보다 먼저 기준 문서와 핵심 계약을 다시 맞추고, `convert` 분리, 기본 검증 공통화, 서비스 한글 주석 및 로그 정리, Mapper alias 정리처럼 반복 구조를 묶어 정리하는 작업이 적절하다.

코드 한두 줄의 버그보다 "운영 판단 기준이 되는 문서"와 "실제 동작"이 다르다는 점, 그리고 반복 로직이 서비스 내부에 누적된 점이 더 큰 리스크다.
