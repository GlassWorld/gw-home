# Review: 보고관리 제거 및 주간보고 역할 재정렬

> 작업 레벨: **HEAVY**
> 관련 review: `review/work/02-daily-report.md`
> 관련 todo: 미생성

---

## 작업 개요

현재 상단 메뉴에는 `일일보고`, `주간보고`, `보고관리`가 함께 존재한다.
하지만 운영 기준은 이미 다음처럼 정리되어 있다.

- 일일보고는 개인이 작성/수정/조회하는 개인 기록
- 주간보고는 여러 사람이 조회하고 점검할 수 있는 공유 보고

이 기준이라면 `보고관리` 메뉴는 역할이 중복되거나 잘못된 기대를 만든다.
특히 현재 `보고관리`는 관리자용 `일일보고 관리` 화면으로 연결되는데, 이는 \"관리\"의 중심이 일일보고에 있는 것처럼 보이게 만든다.

따라서 이번 검토의 핵심은 다음 두 가지다.

1. 상단 정보구조에서 `보고관리`를 제거할지
2. 팀 조회/점검 흐름을 `주간보고` 중심으로 다시 설계할지

---

## 변경 목적

- 메뉴 구조를 실제 운영 기준과 일치시키기
- 일일보고와 주간보고의 책임 범위를 다시 명확히 하기
- 관리자/리더의 점검 흐름을 `일일보고 관리`가 아니라 `주간보고 조회/점검` 쪽으로 재정렬하기
- 이후 주간보고 개선 작업의 기준 문서를 먼저 고정하기

---

## 현재 구조와 문제

### 1. 메뉴 명칭과 실제 역할이 어긋난다

현재 헤더는 다음 구조다.

- 업무등록
- 일일보고
- 주간보고
- 보고관리

이 중 `보고관리`는 관리자에게만 보이지만, 연결 대상은 `admin/daily-reports`다.
즉 명칭은 넓은 의미의 \"보고\"인데 실제 기능은 `일일보고 운영 조회`다.

문제:

- 사용자는 `보고관리`가 일일/주간 전체 관리인지 이해하기 어렵다.
- 관리자도 \"왜 주간보고는 별도 메뉴고, 보고관리는 일일보고인가\"라는 혼선을 겪기 쉽다.
- 현재 합의된 운영 기준과 UI 정보구조가 불일치한다.

### 2. 일일보고와 주간보고의 공개 범위가 다르다

기존 검토 기준상:

- 일일보고는 개인 기록
- 주간보고는 공유 가능한 보고

그런데 현재 관리자 메뉴가 `일일보고 관리` 쪽에만 드러나 있으면,
운영상 점검과 공유의 중심이 일일보고처럼 보인다.
이는 데이터의 성격과 화면 구조를 뒤섞는다.

### 3. 현재 주간보고 화면은 \"개인 작성기\" 성격이 강하다

현재 `gw-home-ui/pages/work/weekly-reports/index.vue` 기준으로 주간보고는:

- 본인 저장 목록
- 본문 편집
- 같은 주의 일일보고 참고
- AI 초안 생성

흐름에 집중되어 있다.

즉 현재 주간보고는 \"공유/조회/점검\"보다 \"내가 작성하는 편집기\"에 더 가깝다.
이 상태에서는 `보고관리` 메뉴를 제거하더라도, 주간보고가 곧바로 관리자/리더 점검 화면 역할을 대신하기 어렵다.

---

## 판단 기준

### 1. `보고관리` 메뉴는 제거하는 편이 맞다

현재 합의된 역할 정의상 `보고관리`는 정보구조를 흐린다.
특히 메뉴 수준에서 `일일보고`, `주간보고`, `보고관리`를 병렬로 두는 구조는 다음 오해를 만든다.

- 일일보고도 공유/관리 대상처럼 느껴짐
- 주간보고와 보고관리의 차이가 모호함
- 개인 작성과 운영 점검의 경계가 흐려짐

따라서 메뉴 관점에서는 `보고관리` 제거가 타당하다.

### 2. 관리자 일일보고 화면은 \"메뉴 제거\"와 \"기능 제거\"를 분리해서 봐야 한다

`admin/daily-reports` 화면은 현재 다음 기능을 가진다.

- 기간별 전체 일일보고 조회
- 누락 현황 확인
- 작성자별 리스트 탐색

이 기능 자체는 운영상 가치가 있다.
다만 이 기능을 최상위 헤더에 `보고관리`라는 이름으로 노출하는 것이 문제다.

즉 1차 판단:

- 헤더 메뉴에서는 제거
- 기능 자체는 당장 삭제하지 않음
- 필요 시 관리자 전용 보조 화면 또는 후속 메뉴 구조에서 재배치

이 접근이 안전하다.

### 3. 이후 운영 중심은 `주간보고`로 재설계해야 한다

현재 기준대로라면 여러 사람이 보고 조회/점검하는 대상은 주간보고다.
따라서 다음 개선의 중심은 `주간보고`가 되어야 한다.

필요한 방향 예시:

- 주차 기준 목록 조회 강화
- 작성자 기준 필터
- 공개 여부/확정 여부 상태
- 팀 단위 또는 작성자 단위 점검 화면
- 읽기 전용 상세 흐름 강화
- 개인 편집기와 공유 조회 화면의 분리

즉 주간보고는 더 이상 단순 편집 페이지가 아니라,
\"개인 작성 + 팀 공유 + 점검\"을 모두 수용할 수 있는 구조로 다시 잡아야 한다.

## 권장 방향

### 1차

- 상단 헤더에서 `보고관리` 메뉴 제거
- `admin/daily-reports` 기능은 당장 삭제하지 않고 유지
- 주간보고를 향후 \"공유/조회/점검\"의 중심 메뉴로 명시

### 2차

- 주간보고 화면을 개인 편집 중심에서 조회/점검 가능한 구조로 개편
- 읽기 전용 상세 흐름 추가
- 작성자, 주차, 공개 여부 기준 필터 정리
- 필요 시 관리자/리더 관점의 주간보고 목록 API 추가

### 3차

- 관리자 일일보고 화면의 최종 위치를 재결정
- 완전 비노출 유지 / 관리자 하위 메뉴 이동 / 주간보고 보조지표로 축소 여부 판단

---

## 구현 시 확인할 쟁점

- `보고관리` 제거가 헤더만 대상인지, 전체 라우팅/진입점 정리까지 포함하는지
- `admin/daily-reports`는 그대로 유지할지, 관리자 메뉴 체계 안으로 이동할지
- 주간보고를 `개인 작성`과 `공유 조회` 한 화면에서 다룰지, 편집/상세를 분리할지
- 주간보고 조회 권한이 관리자 전용인지, 일반 사용자도 서로 볼 수 있는지
- 주간보고의 공개 개념을 계속 `openYn` 으로 유지할지, 별도 상태 모델이 필요한지

## 작업 분류 판단

**HEAVY**

사유:

- 단순 메뉴 텍스트 수정이 아니라 보고 체계의 정보구조 재정의에 해당
- 관리자 진입 구조와 주간보고 역할 변경이 함께 연결됨
- 후속으로 주간보고 조회/점검 기능 확장이 거의 확정적임

단, 실제 구현은 단계 분리가 가능하다.

- `보고관리` 메뉴 제거만 하면 LIGHT
- `주간보고`를 점검용 구조로 재설계하면 HEAVY

따라서 다음 계획 문서에서는 `1차 메뉴 정리`와 `2차 주간보고 개편`을 분리하는 것이 적절하다.

## 결론

현재 합의된 운영 기준을 따르면 `보고관리` 메뉴는 유지 이유가 약하다.
개인 기록은 `일일보고`, 공유/조회/점검은 `주간보고`가 맡는 구조가 더 자연스럽다.

따라서 우선순위는 다음이 적절하다.

1. `보고관리` 메뉴 제거
2. 관리자 일일보고 기능은 당장 삭제하지 않고 유지
3. 주간보고를 실제 조회/점검 중심 구조로 별도 개편

즉 이번 이슈의 본질은 메뉴 삭제 자체보다,
보고 체계의 중심을 `일일관리`에서 `주간공유`로 옮기는 재정렬 작업으로 보는 것이 맞다.
